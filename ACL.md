# Access Control List (ACL) 실습

## 개념 정리

### ACL이란
- 네트워크 장비에서 원하지 않는 트래픽을 필터링하여 차단하는 접근 제어 목록
- 라우터 및 Layer 3 장비를 경유하는 모든 패킷에 대해 허용(permit) 또는 차단(deny)을 적용할 수 있음
- 라우팅 테이블 업데이트 시 특정 경로 정보를 전달하지 않도록 제어할 때도 사용

### ACL 동작 원리
- ACL은 위에서 아래로 순서대로 검사하며, 조건에 일치하는 항목이 나오면 해당 동작을 수행하고 나머지는 검사하지 않음
- 모든 ACL의 마지막에는 암묵적으로 `deny any`가 존재함 (명시하지 않아도 자동 적용)
- 따라서 허용할 트래픽이 있다면 반드시 permit 구문을 먼저 작성해야 함

### ACL 종류

| 구분 | 번호 범위 | 검사 항목 | 적용 위치 권장 |
|------|----------|----------|--------------|
| Standard ACL | 1 ~ 99 | 출발지 IP만 검사 | 목적지에 가까운 곳 |
| Extended ACL | 100 ~ 199 | 출발지 IP, 목적지 IP, 프로토콜, 포트 번호 | 출발지에 가까운 곳 |

Standard ACL은 출발지 IP만 보기 때문에, 출발지 근처에 적용하면 의도하지 않은 트래픽까지 차단될 수 있다. 그래서 목적지에 가까운 인터페이스에 적용하는 것이 일반적이다.

Extended ACL은 세밀한 조건을 지정할 수 있으므로 출발지에 가까운 곳에 적용해서 불필요한 트래픽을 빨리 걸러내는 것이 효율적이다.

### ACL 적용 방향

| 방향 | 의미 |
|------|------|
| in | 인터페이스로 들어오는 패킷에 적용 |
| out | 인터페이스에서 나가는 패킷에 적용 |

하나의 인터페이스에 in/out 각각 하나씩, 총 2개의 ACL을 적용할 수 있다.

### Wildcard Mask
- 서브넷 마스크와 반대 개념으로, 0은 일치해야 하는 비트, 1은 무시하는 비트를 의미
- 예: 서브넷 마스크 255.255.255.0 → 와일드카드 마스크 0.0.0.255

| 와일드카드 마스크 | 의미 |
|-----------------|------|
| 0.0.0.0 | 정확히 해당 IP 1개 (host와 동일) |
| 0.0.0.255 | /24 대역 전체 |
| 0.0.255.255 | /16 대역 전체 |
| 255.255.255.255 | 모든 IP (any와 동일) |

## Standard ACL 문법

### 기본 구조
```
access-list [번호] [permit|deny] [출발지 IP] [와일드카드 마스크]
```

### 예시
```
access-list 1 permit 192.168.10.0 0.0.0.255
```
- access-list: ACL 생성 명령
- 1: Standard ACL 번호 (1~99)
- permit: 허용
- 192.168.10.0: 출발지 IP
- 0.0.0.255: 와일드카드 마스크

출발지가 192.168.10.0/24 대역이면 허용한다는 의미.

### 축약 표현

```
access-list 1 deny host 192.168.10.10
access-list 1 permit any
```

| 표현 | 의미 |
|------|------|
| host 192.168.10.10 | 정확히 해당 IP 1개 (0.0.0.0과 동일) |
| any | 모든 IP (255.255.255.255와 동일) |

### 인터페이스에 적용
```
interface [인터페이스]
 ip access-group [ACL 번호] [in|out]
```

## Extended ACL 문법

### 기본 구조
```
access-list [번호] [permit|deny] [프로토콜] [출발지 IP] [와일드카드] [목적지 IP] [와일드카드] [eq 포트번호]
```

### 주요 프로토콜 및 포트 번호

| 프로토콜 | 포트 | 서비스 |
|---------|------|--------|
| tcp | 80 | HTTP (웹) |
| tcp | 443 | HTTPS |
| tcp | 23 | Telnet |
| tcp | 21 | FTP |
| icmp | - | Ping |
| ip | - | 모든 IP 트래픽 |

## 실습 토폴로지

```
  [PC1]  [PC2]                          [SERVER-A]  [SERVER-B]
    |      |                             172.16.10.100  172.16.10.200
    +------+                                  |            |
       |                                      +------+-----+
       | 192.168.10.0/24                             |
       |                                             | 172.16.10.0/24
  +----+-----+        10.0.0.0/30        +----------+--+
  |   R1     |---------------------------|     R2       |
  |  f0/0    | f0/1                f0/1  |    f0/0      |
  +----------+                           +-------------+
  192.168.10.1  10.0.0.1          10.0.0.2   172.16.10.1
```

### IP 주소 구성

**라우터**

| 라우터 | 인터페이스 | IP 주소 | 설명 |
|--------|-----------|---------|------|
| R1 | f0/0 | 192.168.10.1/24 | PC측 LAN |
| R1 | f0/1 | 10.0.0.1/30 | R1-R2 연결 |
| R2 | f0/1 | 10.0.0.2/30 | R1-R2 연결 |
| R2 | f0/0 | 172.16.10.1/24 | 서버측 LAN |

**단말**

| 장비 | IP | Gateway |
|------|----|---------|
| PC1 | 192.168.10.10/24 | 192.168.10.1 |
| PC2 | 192.168.10.11/24 | 192.168.10.1 |
| SERVER-A | 172.16.10.100/24 | 172.16.10.1 |
| SERVER-B | 172.16.10.200/24 | 172.16.10.1 |

## 1단계. 인터페이스 IP 설정

**R1**
```
conf t
interface f0/0
 ip address 192.168.10.1 255.255.255.0
 no shutdown

interface f0/1
 ip address 10.0.0.1 255.255.255.252
 no shutdown
```

**R2**
```
conf t
interface f0/1
 ip address 10.0.0.2 255.255.255.252
 no shutdown

interface f0/0
 ip address 172.16.10.1 255.255.255.0
 no shutdown
```

설정 후 `show ip int brief`로 인터페이스 상태 확인.

## 2단계. 라우팅 설정

ACL 실습 전에 양쪽 네트워크 간 통신이 가능해야 한다. 정적 라우팅 또는 동적 라우팅을 설정.

**R1 (정적 라우팅)**
```
ip route 172.16.10.0 255.255.255.0 10.0.0.2
```

**R2 (정적 라우팅)**
```
ip route 192.168.10.0 255.255.255.0 10.0.0.1
```

PC1에서 SERVER-A로 ping 테스트:
```
PC1> ping 172.16.10.100
```
응답이 오면 기본 통신 확인 완료.

## 3단계. 실습 시나리오 1 - 단일 서버 웹 접근 허용

### 조건
- 출발지: 192.168.10.0/24
- 목적지: SERVER-A (172.16.10.100)의 웹 서비스(포트 80)만 허용
- 그 외 트래픽은 전부 차단

### ACL 설정 (R2)

```
conf t
access-list 100 permit tcp 192.168.10.0 0.0.0.255 host 172.16.10.100 eq 80
access-list 100 deny ip any any

interface f0/0
 ip access-group 100 out
end
```

한 줄씩 설명:
- `permit tcp ... eq 80`: 192.168.10.0/24에서 172.16.10.100으로 가는 TCP 80번(HTTP) 트래픽만 허용
- `deny ip any any`: 나머지 모든 트래픽 차단 (암묵적으로 존재하지만 명시적으로 작성하는 것이 관리상 좋음)
- `ip access-group 100 out`: f0/0에서 나가는 방향에 ACL 100 적용

### 동작 확인

```
PC1에서 SERVER-A 웹 접속 (포트 80)  → 성공
PC1에서 SERVER-A ping              → 차단 (ICMP는 허용하지 않았으므로)
PC1에서 SERVER-B 웹 접속           → 차단
```

### ACL 적용 결과 확인
```
show access-lists
```
각 항목 옆에 매칭 횟수(matches)가 표시되어 실제로 필터링이 동작하는지 확인할 수 있다.

## 4단계. 실습 시나리오 2 - 복수 서버 웹 접근 허용

시나리오 1의 ACL을 삭제하고 새로 설정한다.

### 기존 ACL 삭제
```
conf t
no access-list 100

interface f0/0
 no ip access-group 100 out
end
```

### 조건
- 출발지: 192.168.10.0/24
- 목적지: SERVER-A (172.16.10.100)와 SERVER-B (172.16.10.200)의 웹 서비스만 허용
- 그 외 트래픽은 전부 차단

### ACL 설정 (R2)

```
conf t
access-list 100 permit tcp 192.168.10.0 0.0.0.255 host 172.16.10.100 eq 80
access-list 100 permit tcp 192.168.10.0 0.0.0.255 host 172.16.10.200 eq 80
access-list 100 deny ip any any

interface f0/0
 ip access-group 100 out
end
```

ACL은 위에서 아래로 순서대로 검사하기 때문에, permit 조건을 먼저 쓰고 마지막에 deny로 나머지를 막는 구조다.

### 동작 확인

```
PC1에서 SERVER-A 웹 접속 (포트 80)  → 성공
PC1에서 SERVER-B 웹 접속 (포트 80)  → 성공
PC1에서 SERVER-A ping              → 차단
PC1에서 SERVER-B ping              → 차단
```

## ACL 확인 및 관리 명령어

```
show access-lists              -- 설정된 ACL 목록과 매칭 횟수 확인
show ip interface [인터페이스]  -- 해당 인터페이스에 적용된 ACL 확인
show running-config            -- 전체 설정에서 ACL 관련 부분 확인
```

### ACL 삭제

```
no access-list [번호]          -- 해당 번호의 ACL 전체 삭제
```

Standard/Extended ACL은 번호 기반이라 개별 항목만 삭제할 수 없고, 전체를 지운 뒤 다시 작성해야 한다. 개별 항목 수정이 필요하면 Named ACL을 사용하는 것이 편리하다.

## 패킷 흐름 정리

```
PC1 (192.168.10.10)
  |
  | 1. PC1이 SERVER-A 웹 페이지 요청 (TCP 80)
  v
[R1 f0/0] --- 수신 ---> [R1 f0/1] --- 전달 ---> [R2 f0/1] --- 수신
                                                    |
                                          2. R2가 라우팅 테이블 확인
                                          3. f0/0 out 방향에 ACL 100 적용됨
                                          4. ACL 검사 시작
                                             ├─ 1번 항목: permit tcp ... 100 eq 80 → 일치 → 허용
                                             ├─ 2번 항목: permit tcp ... 200 eq 80 → (검사 안 함)
                                             └─ 3번 항목: deny ip any any → (검사 안 함)
                                                    |
                                          5. 패킷 통과
                                                    v
                                              [R2 f0/0] ---> SERVER-A (172.16.10.100)
```

---

## 정리

- Standard ACL은 출발지 IP만 검사하고, 목적지 가까이에 적용하는 것이 원칙
- Extended ACL은 출발지, 목적지, 프로토콜, 포트까지 세밀하게 제어 가능하고, 출발지 가까이에 적용
- ACL은 위에서 아래로 순서대로 매칭되며, 마지막에 암묵적 deny any가 존재
- 허용할 트래픽을 먼저 permit으로 작성하고, 마지막에 deny로 나머지를 차단하는 것이 일반적인 구성 방식
- ACL 적용 후 `show access-lists`로 매칭 횟수를 확인하면서 의도대로 동작하는지 검증하는 습관이 중요
